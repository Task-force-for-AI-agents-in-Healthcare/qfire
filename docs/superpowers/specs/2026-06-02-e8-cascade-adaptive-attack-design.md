# E8 — Cascade adaptive attack (3-stage) — Design

**Date:** 2026-06-02
**Backlog:** [paper-strengthening backlog](2026-06-01-paper-strengthening-experiments-backlog.md)
**Status:** DRAFT — pending user review before the implementation plan.

## Research question

E1 stress-tested QFIRE with three *adaptive* families (scope-impersonation,
paraphrase-to-evade, encoding/suffix), generated by `gemma2:9B`. The tool-boundary-firewall
work~\cite{firewallsbench2026} argues this is still not enough: they show static and
single-shot adaptive benchmarks are "easily saturated," and propose a **3-stage cascade**
— standard → second-order → adaptive — as the protocol that actually exposes a defense's
robustness. E8 applies that staged protocol to QFIRE: *does the positive-security
scope+PHI chain hold as attacks escalate from canonical, to defense-aware second-order,
to fully QFIRE-adaptive?*

## What's reused

- The E1 corpora + generators: `gen_adaptive_attacks.py`, `paraphrase_evade.py`
  (the query-until-allowed loop), `make_encoded_sets.py`.
- The deployed chains (`hipaa_phi` for healthcare, `default` for injection) scored via
  `qfire bench --no-cache`, and `analyze_adaptive.py`/`plot_adaptive.py`.
- The panel detectors (DeBERTa, PromptGuard-2, Sentinel) for side-by-side staged recall.

## Method — the 3-stage cascade (recall reported at each stage)

1. **Stage 1 — standard.** Canonical injection / PHI-exfil / scope-drift payloads
   (the existing static + E1 family-0 set). Establishes the baseline block rate.
2. **Stage 2 — second-order (defense-aware).** Attacks rewritten with knowledge of the
   *class* of defense (positive-security scope + PHI matching): e.g. payloads phrased to
   satisfy a plausible declared scope, identifier-free PHI requests, instruction payloads
   deferred/embedded so the triggering content looks in-scope. Generated by a local LLM
   prompted with QFIRE's rule semantics (not its verdicts).
3. **Stage 3 — adaptive (QFIRE-in-the-loop).** Generalize the E1 paraphrase loop beyond
   DeBERTa: iteratively mutate each Stage-2 survivor and **query QFIRE's actual chain**
   (`--no-cache`) until it ALLOWs or an iteration budget is hit; report **evasion rate**
   and **median iterations-to-evade** against the *full QFIRE chain* (not just a single
   classifier). This is the strongest, most honest stress test.

Report a **staged recall curve** (Stage 1→2→3) for QFIRE scope+PHI vs the classifier
panel, plus the Stage-3 evasion rate / iteration count. Honest-negative is the explicit
goal: if Stage 3 finds a reliable QFIRE evasion, that is a primary result (and a fix
target), not a failure to hide.

## Deliverables

- `scripts/gen_cascade_attacks.py` (Stage 2 generator) and a `scripts/adapt_vs_qfire.py`
  (Stage 3 loop: mutate → `qfire check`/`bench --no-cache` → repeat), reusing the
  `evade_loop`/`is_blocked` structure from `paraphrase_evade.py`.
- Staged corpora under `corpora/adaptive/cascade_{stage2,stage3}/`.
- A staged-recall figure + an evasion-rate/iterations table; `analyze_adaptive.py`
  extended (or a sibling analyzer) for the staged view.
- Paper: extend §Adaptive with the cascade result; cite `firewallsbench2026` as the
  protocol source. Findings doc; backlog E8 ticked.

## Success criterion

A Stage 1→2→3 recall curve for QFIRE vs classifiers and a Stage-3 (QFIRE-adaptive)
evasion rate + median iterations. Either outcome is publishable: QFIRE holding through
Stage 3 strengthens the robustness claim; a Stage-3 evasion is reported plainly with the
attack shape and a discussion of the fix (e.g. tightening scope rules, adding a detector
column).

## Resolved decisions (user, 2026-06-02)
1. Full design spec now (this doc); implementation plan when picked up.
2. Adaptive attacks remain **local-LLM-generated** (seed 42, generator reported), a lower
   bound on a human red-teamer — consistent with E1's framing.
3. **Stage-3 iteration budget:** **fixed N = 10 mutations/attack** (matches the E1
   paraphrase loop; predictable Ollama cost). Report the cap; an attack unevaded at N=10
   counts as blocked.
4. **Stage-2 generator:** the **strongest available local model** (e.g. gpt-oss:20b /
   gemma3:27b), not E1's `gemma2:9B`. This makes the defense-aware rewrites a *harder,
   more honest* test; note explicitly that E8 is therefore a **strictly stronger** probe
   than E1 and is **not generator-matched** to it (so E8 numbers are a tougher bar, not a
   like-for-like delta vs E1).
5. **Granularity:** report Stage-3 against the **full** `hipaa_phi`/`default` chain **and
   per-component** (scope-judge-only, PHI-only) to show which component the adaptive
   pressure targets/breaks.

## Feasibility & risks
- **Medium**, reuses E1 infra. The Stage-3 loop queries QFIRE per mutation → Ollama-bound
  (the scope judge runs each query); budget accordingly and `--no-cache` is mandatory
  (cache key omits scope text → rungs would collude).
- **Generator quality:** Stage-2 "defense-aware" rewrites must preserve malicious intent
  (E1 saw `gemma2` dilute injection into benign-ish text); reuse the E1 mitigation
  (healthcare carries a hard PHI identifier; keep intent-preserving paraphrase for
  injection) and spot-check a sample for intent retention.
- **Iteration budget:** cap Stage-3 iterations (report the cap; no silent truncation).

## Open questions — RESOLVED (2026-06-02)
1. Stage-3 budget → fixed N=10 mutations/attack (decision 3).
2. Stage-2 generator → strongest local model; E8 is a strictly harder, non-generator-matched
   probe vs E1 (decision 4).
3. Granularity → full chain + per-component (decision 5).
